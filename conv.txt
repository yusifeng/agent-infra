### 1. 不是上下位替代关系

不是“你做了 `agent-infra`，就不需要 `pi-mono`”；

### 2. 是“部分重叠、但分层不同”的关系

重叠点在：

* message/state
* tool execution
* agent loop
* session/thread 概念

不同点在：

* `pi-mono` 更偏**runtime + UI/toolkit**
* 你的 `agent-infra` 更偏**backend infra + persistence**

### 3. 实际上可以组合

最自然的组合方式其实是：

* 用 `pi-agent-core` 作为 **runtime adapter**
* 用你的 `agent-infra` 作为 **durable backend**
* 把 `pi-agent-core` 的事件流、message/tool 状态映射到你的

  * `run`
  * `message`
  * `message_part`
  * `tool_invocation`
  * 未来的 `run_events`

因为 `pi-agent-core` 已经有非常清晰的事件序列：`agent_start`、`turn_start`、`message_update`、`tool_execution_start/end` 等，这些事件天然适合落到你未来想做的 event log 里。

如果你让我用一句话下定义：

**`pi-mono` 像“可直接拿来跑 agent 的框架和工具箱”，而你的 `agent-infra` 像“让这些 agent 可持久化、可回放、可审计、可替换 runtime 的后端底座”。**

所以你后面如果继续做 infra，最有价值的方向不是去和 `pi-mono` 正面重合做 UI/CLI，而是：

* 做更强的 `run` 状态机
* 做 `run_events`
* 做 runtime adapter 层
* 让 `pi-agent-core`、OpenAI Agents、LangGraph 这类 runtime 都能挂到你的数据底座上

这样定位会更清楚。

## 3. 两者关系

最准确的说法是：

**`pi-mono` = 更靠上层的 agent toolkit**
**`agent-infra` = 更靠底层的 execution/persistence infra**

它们有重叠，但不是一回事。

### `pi-mono` 已经明显实现了的

这些你大概率**不用自己再做一遍**：

* 多 provider 模型抽象：`pi-ai`。
* Agent runtime：`Agent`、`prompt()`、`continue()`、`abort()`、tool execution、parallel/sequential tools。
* 事件流：`agent_start / turn_start / message_update / tool_execution_start / tool_execution_end ...`。
* Web chat UI：`ChatPanel`、`AgentInterface`、artifacts、attachments。
* 前端会话存储：IndexedDB 的 sessions / settings / provider keys。

如果你要做的是：

* coding agent
* browser chat app
* 带 artifacts 的 web 界面
* 快速起一个 agent 产品

那我会很认真地建议你：**优先用它，不要重复造轮子。**

## 4. `pi-mono` 从 README 看起来没替你做全的

这部分才是你要不要继续做 `agent-infra` 的关键。

从它的 README 里，我**能明确看到**的是浏览器端的 IndexedDB sessions/store；但我**没有看到**它提供一套像你现在这样明确的、后端数据库中心的：

* `thread / run / message / message_part / tool_invocation`
* Postgres/SQLite 双路径持久化
* 服务端可审计执行记录
* runtime-agnostic repository layer
* durable run replay / audit log / admin trace

`pi-web-ui` 的 storage 重点是前端应用存储，不是你现在做的这种服务端 infra schema。

`pi-agent-core` 很强，但它更像**运行时**，不是你这种**durable backend storage contract**。它讲了 message flow、event flow、tool lifecycle、Agent state，但不是以数据库 schema 为核心来设计的。

所以如果你的真实目标是下面这些：

* 同一套后端支持多个不同 runtime
* 所有 run/tool/message 都能入库审计
* 后续做 replay / observability / timeline
* 不想绑死在某个 agent framework
* 想把 thread/run/message 变成稳定业务真相源

那你自己的 `agent-infra` 仍然有意义。

## 5. 我给你的直接建议

### 情况 A：你主要想“做出产品”

那就：

**直接用 `pi-mono`**

尤其如果你想要的是：

* agent runtime
* chat UI
* tool 展示
* attachments / artifacts
* 快速成型

这时你再自己实现一套 runtime/UI，收益不高。 

### 情况 B：你主要想“做底座”

那就：

**继续保留 `agent-infra`，但别和 `pi-mono` 正面重复**

而是把它定位成：

* runtime-agnostic persistence layer
* execution record backend
* thread/run/message durable store
* future replay / trace / admin infra

然后以后甚至可以：

**用 `pi-agent-core` 当 runtime，接到你的 `agent-infra` 上。**

也就是：

```text id="14390"
pi-agent-core
   ↓ emits events / state changes
agent-infra adapter
   ↓ persists
thread / run / message / tool_invocation / run_events
```

这个组合是成立的。
