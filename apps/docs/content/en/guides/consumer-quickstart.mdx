# Consumer Quickstart

This guide shows the intended way to consume `agent-infra` in an app.

The short version is:

1. keep durable truth in packages
2. call `@agent-infra/app` from your server boundary
3. treat `@agent-infra/runtime-pi` as a runtime adapter, not as your product API
4. use timeline reads and SSE as read paths, not as the only source of truth

## What a consumer should own

A consumer app should own:

- HTTP or RPC routes
- auth and tenancy rules
- app-specific page composition
- small UI-only state such as panel selection or draft input

A consumer app should **not** own:

- core `thread`, `run`, `message`, `message_part`, `tool_invocation`, or `run_event` shapes
- direct local redefinitions of durable records
- agent loop orchestration hidden inside React components

## Package roles

The intended package split is:

- `@agent-infra/core`
  - domain types and repository interfaces
- `@agent-infra/db`
  - SQLite and PostgreSQL repository implementations
- `@agent-infra/runtime-pi`
  - runtime adapter that persists run lifecycle truth
- `@agent-infra/app`
  - narrow use-case boundary for consumers
- `@agent-infra/contracts`
  - serialized DTOs for HTTP and SSE

The key rule is simple:

`@agent-infra/app` is the boundary your consumer should call.

## Minimal server wiring

Most consumers need four things:

1. a DB config
2. repository instances
3. a runtime port
4. one `createAgentInfraApp(...)` instance

The reference shape looks like this:

```ts
import crypto from 'node:crypto';

import {
  createAgentInfraApp,
  type AgentInfraAppDependencies,
  type AgentInfraAppRepositories,
  type AgentInfraRuntimePort
} from '@agent-infra/app';
import {
  createDbConfigFromEnv,
  DrizzleMessageRepository,
  DrizzleRunEventRepository,
  DrizzleRunRepository,
  DrizzleThreadRepository,
  DrizzleToolInvocationRepository,
  SqliteMessageRepository,
  SqliteRunEventRepository,
  SqliteRunRepository,
  SqliteThreadRepository,
  SqliteToolInvocationRepository,
  withDbTransaction,
  type DbConfig
} from '@agent-infra/db';
import { createPiRuntime } from '@agent-infra/runtime-pi/runtime';
import { createDemoTools } from '@agent-infra/runtime-pi/tools';

function createRepositories(dbConfig: DbConfig, db: any): AgentInfraAppRepositories {
  if (dbConfig.mode === 'sqlite') {
    return {
      threadRepo: new SqliteThreadRepository(db),
      runRepo: new SqliteRunRepository(db),
      messageRepo: new SqliteMessageRepository(db),
      toolRepo: new SqliteToolInvocationRepository(db),
      runEventRepo: new SqliteRunEventRepository(db)
    };
  }

  return {
    threadRepo: new DrizzleThreadRepository(db),
    runRepo: new DrizzleRunRepository(db),
    messageRepo: new DrizzleMessageRepository(db),
    toolRepo: new DrizzleToolInvocationRepository(db),
    runEventRepo: new DrizzleRunEventRepository(db)
  };
}

const dbConfig = createDbConfigFromEnv();
await dbConfig.initialize();

const runtime = createPiRuntime({
  tools: (context) => createDemoTools(context)
});

const runtimePort: AgentInfraRuntimePort = {
  async prepare(preferred) {
    return runtime.prepare(preferred);
  },
  async runTextTurn(repositories, input) {
    await runtime.runTurn(
      {
        runRepo: repositories.runRepo,
        messageRepo: repositories.messageRepo,
        toolRepo: repositories.toolRepo,
        runEventRepo: repositories.runEventRepo
      },
      input
    );
  }
};

const repositories = createRepositories(dbConfig, dbConfig.db);

const app = createAgentInfraApp({
  repositories,
  transaction: async (operation) =>
    withDbTransaction(dbConfig, async (tx) => operation(createRepositories(dbConfig, tx))),
  runtime: runtimePort,
  idGenerator: () => crypto.randomUUID(),
  now: () => new Date()
} satisfies AgentInfraAppDependencies);
```

This guide intentionally uses the `@agent-infra/runtime-pi/runtime` and `@agent-infra/runtime-pi/tools` subpaths.

- the root `@agent-infra/runtime-pi` entrypoint is kept for types and config helpers
- runtime execution helpers require an explicit heavy import path
- that keeps Next.js-style server module graphs less likely to pull runtime dependencies into read paths or first-load paths by accident

If you want a working repository example, start from `apps/playground-web/lib/playground-services.ts`.

## Consumer request flow

For a normal text turn, the recommended flow is:

1. `app.threads.create(...)` or `app.threads.list(...)`
2. `app.threads.getMessages(...)` for transcript reads
3. `app.turns.startText(...)` or `app.turns.runText(...)` to queue and execute a turn
4. `app.runs.listByThread(...)` for recent run inspection
5. `app.runs.getTimeline(...)` for run logs and tool traces

This keeps the consumer API small and stable.

## Transcript, timeline, and SSE

Treat the three read paths differently:

- transcript
  - durable conversation projection
  - read from `threads.getMessages(...)`
- timeline
  - durable run trace
  - read from `runs.getTimeline(...)`
- SSE
  - live transport for already-persisted or in-flight updates
  - useful for running consoles, not the only truth

For `v0`, this means:

- final assistant content settles through durable `message_part` writes
- `message_update` can drive a live assistant draft in the UI
- refresh or reconnect should rebuild from durable reads

## UI rules that keep the boundary healthy

If you build a console or product UI on top of `agent-infra`, keep these rules:

- use package types and contracts instead of redefining core records in the app
- keep run logs run-oriented and transcript thread-oriented
- do not treat browser memory as the durable state model
- do not let SSE become the only way to understand what happened
- keep product-specific workflow logic out of the runtime adapter

## Good first features for a consumer

The current `v0`-aligned consumer surface is:

- thread list and thread creation
- transcript rendering from durable messages
- send text turn
- recent run switcher
- run log panel based on timeline plus live SSE
- reload-safe restoration of selected thread and run

That is enough to validate the platform without turning the consumer into a full chat product.
